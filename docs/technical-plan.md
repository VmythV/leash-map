# 领迹 LeashMap 产品技术方案 v1

> 评审版本：v1.0  
> 日期：2026-06-26  
> 依据：`docs/brand.md`

## 1. 方案结论

领迹应按“宠物安全与轨迹可视化系统”建设，而不是只做一个防丢蓝牙标签。产品由一个中成本、低功耗、定位精准的宠物定位器，以及 App、云端、可视化后台组成。

硬件建议以“4G Cat 1 bis + 双频多星座 GNSS + 低功耗运动传感器 + 可 OTA 固件”为主方案，优先服务中国市场和城市户外场景；海外版可保留 LTE-M/NB-IoT SiP 方案作为后续分支。

软件建议一次性按平台化设计：设备数据先进入云端，再由 App 和后台消费同一套位置、轨迹、围栏、告警和设备状态服务。这样 C 端和 B 端可以共用底层能力，后续扩展 `领迹 Fleet / Shelter / Park` 不需要重做核心链路。

## 2. 产品范围

### 2.1 本期包含

- 项圈内置或外挂式宠物定位器。
- 实时定位、历史轨迹、电子围栏、离区提醒、低电提醒、离线提醒。
- 宠物运动距离、活动时长、活跃时段、常去区域。
- 领迹 App：宠物主人使用。
- 领迹 Cloud：设备接入、数据处理、存储、告警、接口。
- 领迹 Vision：后台可视化，多设备、多宠物管理。

### 2.2 本期不包含

- 不承诺“100% 找回”“永不丢失”“零误差定位”。
- 不做宠物医疗诊断。
- 不在 v1 强依赖 RTK 厘米级定位，因为成本、功耗、天线、订阅服务和户外环境复杂度都会上升。
- 不把 AirTag/蓝牙防丢器作为主方案，蓝牙只作为近距离辅助。

## 3. 核心指标

### 3.1 硬件目标

| 指标 | v1 目标 |
| --- | --- |
| 设备形态 | 项圈内置核心模块或轻量外挂模块 |
| 重量 | Core/Tag <= 35g；Mini 目标 <= 25g |
| 防护 | IP67 起步，目标支持雨水、泥水、短时浸水 |
| 电池 | 500-700mAh 锂电池，磁吸触点或隐藏式充电 |
| 正常续航 | 日常模式 3-7 天，取决于定位频率和网络质量 |
| 高精度追踪续航 | 高频寻宠模式 6-12 小时 |
| 待机续航 | 低频上报 + 运动唤醒目标 14 天以上 |
| 定位 | GNSS 优先，蜂窝基站/Wi-Fi Scan/BLE 辅助 |
| OTA | 支持固件远程升级和失败回滚 |

### 3.2 定位目标

| 场景 | 目标表现 |
| --- | --- |
| 开阔户外 | P50 <= 5m，P90 <= 15m |
| 普通城市街区 | P50 <= 15m，P90 <= 50m |
| 室内/地下/强遮挡 | 自动降级为基站/Wi-Fi/BLE 近似定位 |
| 电子围栏提醒 | 室外有效定位后 10-30 秒内触发 |
| 轨迹回放 | 支持按天、时间段、宠物、设备查询 |

说明：定位精度必须按真实宠物佩戴环境验证。宠物身体遮挡、项圈方向、天线面积、楼宇反射都会显著影响 GNSS 表现。

## 4. 硬件方案

### 4.1 推荐硬件架构

```text
电池/充电/电量计
      |
低功耗 MCU 或主控 ---- IMU 运动传感器
      |                    |
      |                    +-- 运动唤醒、静止判断、活动量
      |
4G Cat 1 bis 通信模组 ---- LTE 主天线
      |
      +-- MQTT/HTTPS/TLS 数据上报
      |
双频 GNSS 模组 ----------- GNSS 天线
      |
      +-- GPS/BDS/Galileo/GLONASS/QZSS/NavIC
```

### 4.2 主推方案：国内优先版 Core v1

| 模块 | 建议 |
| --- | --- |
| 蜂窝通信 | Quectel EG915Q / EG800K / EG810M 等 Cat 1 bis 系列，按供应链和认证选型 |
| GNSS | Quectel LC29H 标准精度双频版本，优先 L1+L5、多星座 |
| MCU | STM32U0/STM32L4 或 Nordic nRF52，负责低功耗状态机、传感器、数据缓存 |
| 运动传感器 | ST LIS2DW12 或同级超低功耗三轴加速度计 |
| 电量计 | MAX17048 / BQ274xx 同级 fuel gauge |
| 充电 | 单节锂电充电管理，支持温度保护和过充过放保护 |
| SIM | 量产建议 MFF2 eSIM；早期样机可 nano SIM 降低调试成本 |
| 本地提示 | 蜂鸣器 + 单色/双色 LED，寻宠和设备状态提示 |
| 近距离通信 | BLE 可选，用于配网、靠近提醒、主人手机附近省电 |

选择理由：

- Cat 1 bis 在功耗、成本、移动性、覆盖和实时性之间较平衡，适合宠物这种移动场景。
- 双频 GNSS 比单频更适合城市楼宇反射场景，能降低多路径误差。
- IMU 可让 GNSS 和蜂窝模组按运动状态唤醒，避免持续高功耗定位。
- 设备不要依赖单一定位来源；GNSS 失败时必须降级到蜂窝基站、Wi-Fi Scan 或 BLE 近距状态。

### 4.3 海外低功耗分支

海外版可评估 Nordic nRF9151 这类集成 LTE-M/NB-IoT/NTN/GNSS 的 SiP。它的集成度高，适合低功耗资产追踪，但是否适合作为中国市场主版本，需要先确认运营商网络、资费、认证和移动性体验。

建议策略：

- 中国市场：Cat 1 bis + 独立双频 GNSS。
- 北美/欧洲市场：LTE-M 优先，NB-IoT 作为低频补充。
- 不建议把 NB-IoT 作为宠物实时追踪主链路，因为移动性、时延和频繁上报体验通常不如 Cat 1 bis / LTE-M。

### 4.4 不推荐方案

| 方案 | 不推荐原因 |
| --- | --- |
| 纯蓝牙/AirTag 类 | 依赖附近手机网络，不适合实时轨迹和 B 端后台 |
| 纯 NB-IoT | 移动性和实时性风险高 |
| 持续 GNSS 高频定位 | 续航不可接受 |
| v1 上 RTK | 成本、功耗、天线、订阅、稳定性都不适合首版消费级设备 |
| 只用基站定位 | 精度不足，无法支撑“轨迹可视化”心智 |

### 4.5 功耗策略

设备按场景切换工作模式：

| 模式 | 触发条件 | 定位策略 | 上报策略 |
| --- | --- | --- | --- |
| 静止待机 | 长时间无运动 | GNSS 关闭，IMU 低功耗监听 | 30-60 分钟心跳 |
| 日常活动 | 检测到运动 | 1-5 分钟一次 GNSS 或基站/Wi-Fi 辅助 | 1-5 分钟批量上报 |
| 遛宠模式 | App 手动开启或主人手机靠近 | 30-60 秒一次定位 | 30-60 秒上报 |
| 离区警戒 | 接近或离开围栏 | 10-30 秒一次定位 | 立即上报并触发告警 |
| 寻宠模式 | 用户手动开启 | 5-10 秒一次定位，蜂鸣/LED 可启用 | 5-10 秒实时上报 |
| 低电模式 | 电量低于阈值 | 降低定位频率 | 低频心跳 + 低电提醒 |

### 4.6 粗略 BOM 区间

以下仅用于立项评估，实际价格必须以供应商报价、采购量、认证、天线和结构方案为准。

| 项目 | 估算区间 |
| --- | --- |
| Cat 1 bis 通信模组 | 28-55 元 |
| 双频 GNSS 模组 | 35-90 元 |
| MCU + IMU + 电源管理 | 20-50 元 |
| 电池 + 充电 + 电量计 | 20-45 元 |
| LTE/GNSS/BLE 天线与 RF 辅料 | 15-40 元 |
| PCB、外壳、防水、装配 | 35-80 元 |
| 单机硬件 BOM 目标 | 150-300 元 |

成本优化路径：

- v1 先保证定位和续航，不急于极限小型化。
- Mini 版本再做高集成和专用天线结构。
- 若 Cat 1 bis 模组自带 GNSS 表现达标，可评估取消独立 GNSS 模组，但必须通过实测确认。

## 5. 固件方案

### 5.1 固件职责

- 采集 GNSS、基站、Wi-Fi Scan、BLE、IMU、电量、温度、网络状态。
- 根据运动状态和云端配置切换定位频率。
- 本地缓存离线轨迹，网络恢复后补传。
- 维护电子围栏轻量规则，用于快速本地判断和省电。
- 执行 MQTT/HTTPS 数据上报。
- 支持远程配置、日志采样、FOTA 升级、异常自恢复。

### 5.2 固件状态机

```text
BOOT
  -> PROVISIONING  未绑定/待激活
  -> IDLE          静止低功耗
  -> TRACKING      普通运动追踪
  -> GUARD         围栏警戒
  -> LOST          寻宠高频模式
  -> LOW_BATTERY   低电保护
  -> OTA           固件升级
  -> FAULT_RECOVER 异常恢复
```

### 5.3 数据上报协议

优先 MQTT over TLS，HTTPS 作为兜底。消息格式建议 Protobuf 或 CBOR，减少流量与功耗。

核心上报字段：

```json
{
  "device_id": "lm_core_001",
  "pet_id": "pet_001",
  "ts": 1782432000,
  "lat": 31.2304,
  "lng": 121.4737,
  "accuracy_m": 8.5,
  "source": "gnss",
  "speed_mps": 1.2,
  "heading": 86,
  "battery_pct": 72,
  "network": {
    "type": "cat1bis",
    "rssi": -84,
    "cell_id": "..."
  },
  "motion": {
    "state": "moving",
    "steps": 128
  },
  "event": "location"
}
```

### 5.4 安全与 OTA

- 每台设备出厂写入唯一设备证书或密钥。
- 设备与云端使用 TLS，敏感接口做双向认证。
- 固件包签名校验，升级失败自动回滚。
- 设备端保存最小必要日志，不保存用户敏感文本。
- 远程指令必须有过期时间和幂等 ID，避免重复执行。

## 6. 云端架构

### 6.1 总体架构

```text
定位器
  -> MQTT/HTTPS 接入层
  -> 消息队列
  -> 位置处理服务
  -> 轨迹服务 / 围栏服务 / 告警服务 / 设备服务
  -> PostgreSQL + PostGIS / TimescaleDB / Redis / 对象存储
  -> App API / Vision API / WebSocket 实时推送
```

### 6.2 核心服务

| 服务 | 职责 |
| --- | --- |
| Device Service | 设备注册、绑定、证书、在线状态、电量、固件版本 |
| Location Ingest | 接收设备定位点，校验、去重、写入队列 |
| Track Service | 轨迹查询、压缩、回放、导出、热力统计 |
| Geofence Service | 围栏配置、空间计算、进出判断、防误报 |
| Alert Service | 离区、低电、离线、异常活动、回家提醒 |
| Pet/User Service | 用户、宠物、家庭成员、权限管理 |
| OTA Service | 固件版本、灰度发布、升级状态、失败回滚 |
| Notification Service | App Push、短信/电话扩展、站内通知 |
| Vision Service | B 端组织、设备批量管理、地图大屏、告警记录 |

### 6.3 数据存储

| 数据 | 存储建议 |
| --- | --- |
| 用户、宠物、设备、权限 | PostgreSQL |
| 电子围栏、空间查询 | PostgreSQL + PostGIS |
| 高频定位点 | TimescaleDB 或 PostgreSQL 分区表 |
| 最新位置、在线状态 | Redis |
| 告警事件 | PostgreSQL + 可选 Elasticsearch/OpenSearch |
| 轨迹导出、日志包 | 对象存储 |
| 消息解耦 | Kafka / NATS / RabbitMQ，早期可先用 Redis Stream |

### 6.4 定位数据处理

处理流水线：

1. 接入层校验设备身份和消息签名。
2. 丢弃明显非法点：坐标为空、时间倒退、速度异常、精度半径过大。
3. 结合 accuracy、speed、IMU state、network quality 做质量评分。
4. 对轨迹做平滑和抽稀，减少地图锯齿和存储压力。
5. 使用 PostGIS 判断是否进入/离开围栏。
6. 生成告警事件并推送给 App / Vision。

算法建议：

- 轨迹平滑：轻量 Kalman filter。
- 轨迹抽稀：Douglas-Peucker 或基于距离/角度的抽样。
- 常去区域：停留点聚类。
- 热力图：按地理网格或 H3 index 聚合。
- 围栏误报控制：连续 N 个有效点在围栏外，或持续超过 T 秒再提醒。

## 7. App 方案：领迹 App

### 7.1 技术栈

建议使用 Flutter 开发 iOS/Android 双端，地图部分封装原生 SDK：

- 中国大陆：高德地图或腾讯地图。
- 海外：Mapbox 或 Google Maps。
- 实时通道：WebSocket 或 SSE。
- 本地状态：Riverpod/Bloc 同级方案均可。
- 推送：APNs、FCM、国内厂商推送。

### 7.2 核心页面

| 页面 | 功能 |
| --- | --- |
| 地图首页 | 当前宠物位置、设备在线、电量、距离、定位精度 |
| 宠物详情 | 宠物资料、设备绑定、共享成员、最近活动 |
| 轨迹回放 | 按天/时间段回放路径、速度、停留点、距离 |
| 安全区域 | 家、小区、公园围栏设置，离区/回家提醒 |
| 寻宠模式 | 高频定位、蜂鸣器、LED、路线导航 |
| 运动数据 | 日活动距离、活跃时段、趋势、常去区域 |
| 设备管理 | 电量、信号、固件、SIM 状态、充电提醒 |
| 消息中心 | 离区、低电、离线、异常轨迹、回家提醒 |

### 7.3 用户体验原则

- 地图优先，打开 App 先看到宠物位置。
- 提醒文案克制，不制造焦虑。
- 定位精度要透明展示，例如“约 12 米范围内”。
- 当 GNSS 不可用时明确说明“当前为基站/Wi-Fi 近似位置”。
- 寻宠模式必须给用户明显的电量消耗提示。

## 10. 告警策略

### 10.1 离区提醒

避免单点漂移导致误报：

- 只使用 accuracy_m 小于阈值的定位点触发强提醒。
- 连续 2-3 个有效点在围栏外，或持续 20-60 秒在围栏外，再推送离区提醒。
- 如果定位点精度差但趋势显示远离围栏，可先发弱提醒：“位置可能离开安全区域，请确认”。

### 10.2 离线提醒

- 日常模式：超过 15-30 分钟无心跳提醒。
- 寻宠模式：超过 1-3 分钟无实时数据提醒。
- 如果设备处于低电模式，离线阈值自动放宽。

### 10.3 低电提醒

- 30%：建议充电。
- 15%：低电提醒，降低定位频率。
- 5%：进入保底模式，只保留低频心跳和最后位置上报。

## 11. 研发阶段

### P0：技术验证，4-6 周

目标：证明核心链路可行。

- 使用开发板完成 Cat 1 bis / GNSS / IMU 数据采集。
- 完成 MQTT 接入、云端存储、地图实时展示。
- 完成基础轨迹回放和简单圆形围栏。
- 输出第一版功耗、定位、天线测试报告。

### P1：EVT 工程样机，8-10 周

目标：形成可外场测试样机。

- 自研 PCB，完成电源、通信、GNSS、IMU、充电。
- 固件支持状态机、缓存补传、远程配置、基础 OTA。
- App 完成地图首页、设备绑定、轨迹、安全区、提醒。
- Vision 完成多设备地图和告警列表。

### P2：DVT 小批测试，8-12 周

目标：验证结构、天线、防水、续航和稳定性。

- 结构件、防水、佩戴可靠性测试。
- 城市、公园、小区、室内、地下停车场外场测试。
- 100 台级别小批量灰度。
- 完成误报率、漏报率、续航和上传成功率优化。

### P3：PVT 和量产准备，8-12 周

目标：进入量产前准备。

- 供应链锁定和生产测试夹具。
- 设备出厂写证书、IMEI/eSIM 管理、批量激活流程。
- CCC/SRRC/CTA 及目标市场 CE/FCC 等认证计划。
- 客服后台、运维监控、日志诊断、售后换绑流程。

## 12. 验收标准

### 12.1 硬件验收

- 开阔户外连续 30 分钟轨迹无明显断点。
- 城市街区轨迹可读，不出现大面积漂移。
- 日常模式续航达到 3 天以上。
- 寻宠模式连续工作达到 6 小时以上。
- IP67 测试通过。
- 充电、过温、低电保护可靠。

### 12.2 软件验收

- App 可完成设备绑定、实时定位、轨迹回放、围栏设置、告警接收。
- Vision 可查看多设备实时位置、历史轨迹、在线状态和告警记录。
- 云端可支撑至少 10,000 台设备的逻辑架构，早期部署可小规格起步。
- 设备离线后可缓存轨迹，恢复网络后自动补传。
- OTA 支持灰度、失败回滚、升级状态可追踪。

### 12.3 运营验收

- 设备激活流程可控。
- SIM/eSIM 流量用量可统计。
- 设备异常可远程诊断。
- 用户解绑、换绑、丢失、转让流程明确。
- 位置数据支持用户导出和删除。

## 13. 主要风险

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 小体积天线效果不足 | 定位漂移、联网差 | EVT 阶段重点做天线和佩戴姿态测试 |
| 高频定位续航不足 | 用户体验不稳定 | 默认低频，寻宠模式明确提示耗电 |
| 围栏误报 | 用户不信任产品 | 多点确认、精度阈值、弱提醒机制 |
| 室内定位弱 | 无法显示准确位置 | 明确展示定位来源，使用 Wi-Fi/BLE/基站兜底 |
| 运营商网络差异 | 上传延迟或掉线 | 选多运营商卡，云端支持离线补传 |
| 防水与充电冲突 | 返修率高 | 优先磁吸触点或防水结构，减少外露接口 |
| 认证周期 | 上市延期 | P1 后立即启动预检和认证咨询 |

## 14. 待决问题

1. 首发市场是中国大陆还是海外？这会影响 Cat 1 bis、LTE-M/NB-IoT、地图、推送、合规和 SIM 方案。
2. 首发形态是项圈内置 Core，还是外挂 Tag？这会影响天线空间和重量。
3. 目标售价和订阅费区间是多少？这会决定 BOM、流量套餐和云端保留周期。
4. 是否必须支持猫用 Mini？如果必须，重量、电池和天线会成为第一约束。
5. B 端 Vision 是首发能力还是第二阶段？如果首发，权限、组织、批量设备管理需要提前建模。
