# 领迹 LeashMap 最小 MVP 端侧开发划分

> 决策版本：v1.0  
> 日期：2026-06-26  
> 结论：最小 MVP 先做硬件端、固件端、云端、用户 App 四块。

## 1. MVP 目标

最小 MVP 的目标不是一次性完成完整商业化平台，而是先验证领迹的核心闭环：

```text
宠物定位器采集位置
  -> 固件低功耗处理并上报
  -> 云端接收、存储、处理轨迹与告警
  -> 用户 App 查看宠物位置、轨迹和安全提醒
```

只要这条链路成立，就能验证产品最关键的价值：主人可以实时掌握宠物位置变化，查看历史轨迹，并在宠物离开安全区域时收到提醒。

## 2. 首发四个端侧

| 端侧 | MVP 是否必须 | 核心职责 | 必须做的原因 |
| --- | --- | --- | --- |
| 硬件端 | 必须 | 定位器结构、电路、GNSS、4G、天线、电池、防水、充电 | 硬件是位置数据来源。没有稳定、低功耗、精准定位的设备，App 和云端都没有真实业务价值 |
| 固件端 | 必须 | 定位采集、低功耗策略、数据上报、离线缓存、设备状态、OTA 基础能力 | 固件决定设备能否长期稳定运行，并直接影响续航、定位频率、补传能力和远程维护能力 |
| 云端 | 必须 | 设备接入、位置存储、轨迹处理、电子围栏、告警、用户/宠物/设备 API | 云端是设备和 App 之间的中枢，负责把原始定位点变成可查询、可告警、可回放的数据服务 |
| 用户 App | 必须 | 设备绑定、实时位置、轨迹回放、电子围栏、告警通知、寻宠模式 | App 是宠物主人的主要使用入口，直接承载“看见它的每一步”的核心体验 |

## 3. 各端 MVP 边界

### 3.1 硬件端

MVP 先做一款工程样机，不追求完整产品线。

必须包含：

- 4G Cat 1 bis 通信能力。
- GNSS 定位能力，优先双频多星座方案。
- 低功耗运动传感器，用于运动唤醒和活动判断。
- 500-700mAh 电池与基础充电电路。
- 基础防水结构，目标向 IP67 靠近。
- 可外场佩戴测试的项圈安装结构。

暂不包含：

- 猫用极致轻量 Mini 版本。
- 多色外观和完整量产包装。
- RTK 厘米级定位。
- 大规模量产工装和完整认证。

### 3.2 固件端

MVP 固件先保证稳定采集、低功耗和可靠上报。

必须包含：

- GNSS 定位采集。
- 运动状态判断。
- 日常模式、寻宠模式、低电模式。
- MQTT 或 HTTPS 数据上报。
- 网络异常时本地缓存，恢复后补传。
- 电量、信号、设备在线状态上报。
- 基础 OTA 能力，至少支持可控升级和失败保护。

暂不包含：

- 复杂行为识别。
- 宠物健康诊断。
- 多版本复杂灰度策略。
- 高级边缘算法。

### 3.3 云端

MVP 云端先做设备接入和核心位置数据服务。

必须包含：

- 设备注册、绑定、鉴权。
- 定位点接收、校验、存储。
- 最新位置查询。
- 历史轨迹查询。
- 简单轨迹抽稀和平滑。
- 圆形电子围栏。
- 离区提醒、低电提醒、离线提醒。
- App API。
- 基础运维日志和设备状态查看。

暂不包含：

- 完整 B 端组织架构。
- 复杂多门店权限。
- 轨迹热力图高级统计。
- 大屏驾驶舱。
- 复杂开放平台接口。

### 3.4 用户 App

MVP App 先做宠物主人最常用的闭环功能。

必须包含：

- 注册/登录。
- 添加宠物。
- 绑定设备。
- 地图首页查看宠物当前位置。
- 展示设备在线状态、电量、定位精度。
- 查看历史轨迹。
- 设置安全区域。
- 接收离区、低电、离线提醒。
- 开启寻宠模式。

暂不包含：

- 社区功能。
- 宠物健康报告。
- 多人复杂家庭权限。
- 商城和会员体系。
- B 端机构管理。

## 4. 暂缓端侧

| 端侧 | 暂缓原因 | 进入条件 |
| --- | --- | --- |
| 管理端 / 领迹 Vision | B 端价值明确，但不是验证 C 端定位闭环的必要条件 | 当需要服务宠物店、寄养中心、犬舍、宠物乐园时进入 |
| 运维/生产工具端 | MVP 阶段设备数量有限，可先用轻量脚本和云端后台字段支撑 | 进入小批量测试或量产准备阶段时进入 |
| B 端产品线 Fleet / Shelter / Park | 依赖多设备、多宠物、多机构场景，首版会拉高复杂度 | C 端核心链路稳定后再拆分行业版本 |

## 5. MVP 验收标准

MVP 通过标准：

1. 设备可以在真实户外环境持续采集并上报位置。
2. App 可以在地图上看到宠物最新位置。
3. App 可以查询指定时间段的历史轨迹。
4. 用户可以设置一个安全区域。
5. 宠物离开安全区域后，系统可以触发提醒。
6. 设备低电、离线状态可以被 App 感知。
7. 寻宠模式可以提高定位和上报频率。
8. 设备短时离线后可以补传轨迹数据。

## 6. 推荐开发顺序

1. 硬件开发板验证：通信、GNSS、IMU、电池数据。
2. 云端设备接入：先打通定位点上报和存储。
3. App 地图首页：先展示最新位置。
4. 固件低功耗状态机：降低实际功耗。
5. 轨迹回放：形成可视化历史路径。
6. 电子围栏和告警：形成安全闭环。
7. 寻宠模式：形成高价值应急体验。
8. 小规模外场测试：用真实宠物佩戴数据修正定位、功耗和告警策略。

## 7. 与完整方案的关系

本文件是 `docs/technical-plan.md` 的 MVP 执行边界补充。

完整产品仍然保留 `领迹 Vision`、B 端多宠物管理、运维生产工具、更多产品线和高级数据分析能力；但这些能力不进入最小 MVP 的首发范围。
