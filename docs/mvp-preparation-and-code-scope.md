# 领迹 LeashMap MVP 准备清单与代码开发范围

> 决策版本：v1.0  
> 日期：2026-06-26  
> 依据：`docs/mvp-development-scope.md`、`docs/technical-plan.md`

## 1. 结论

基于当前 MVP，首发只做四块：

1. 硬件端
2. 固件端
3. 云端
4. 用户 App

需要准备的硬件，建议先按“开发验证”采购，不要一开始就直接做完整量产硬件。原因是宠物定位器的核心风险在于通信、GNSS 天线、功耗、佩戴姿态和外场稳定性，必须先用开发板和样机把链路跑通。

除了硬件采购之外，可以立即开始写代码的部分很多，尤其是云端、App、设备模拟器、接口协议、轨迹和围栏算法。这些代码不必等真实硬件全部到位。

## 2. 需要准备的硬件

### 2.1 P0：必须先准备的开发验证硬件

这些硬件用于打通 MVP 最小闭环：

```text
定位采集 -> 低功耗处理 -> 蜂窝网络上报 -> 云端接收 -> App 地图展示
```

| 类别 | 建议准备 | 用途 | 说明 |
| --- | --- | --- | --- |
| 4G Cat 1 bis 开发板 | Quectel EG915Q / EG800K / EG810M 同级开发板或评估板 | 验证蜂窝联网、MQTT/HTTPS 上报、SIM 卡稳定性 | 先用官方或代理商评估板，降低硬件调试风险 |
| GNSS 定位开发板 | Quectel LC29H 双频 GNSS 或同级 L1+L5 多星座 GNSS 评估板 | 验证户外定位精度、冷启动、热启动、城市漂移 | 如果 Cat 1 bis 模组自带 GNSS，也建议单独准备 GNSS 板做对比 |
| MCU 开发板 | STM32U0/STM32L4 或 Nordic nRF52 开发板 | 跑固件状态机、IMU 采集、低功耗控制 | MVP 先选团队熟悉的 MCU，不要过早追求极限小型化 |
| 运动传感器板 | ST LIS2DW12 或同级低功耗三轴加速度计模块 | 运动唤醒、静止判断、活动状态判断 | 后续用于省电策略，不建议省略 |
| LTE 天线 | LTE FPC/PCB 天线、IPEX/U.FL 转接线 | 验证 4G 信号与上报稳定性 | 天线位置会直接影响通信稳定性 |
| GNSS 天线 | GNSS 有源/无源天线、IPEX/U.FL 转接线 | 验证定位效果 | 需要测试不同佩戴方向和遮挡情况 |
| IoT SIM 卡 | 中国移动/联通/电信物联网卡或普通测试 SIM | 设备联网和流量测试 | 建议至少准备两个运营商，避免单一网络误判 |
| 锂电池 | 500-700mAh 单节锂电池若干 | 续航测试 | 容量要接近目标产品，否则功耗结论不可靠 |
| 充电与保护模块 | 单节锂电充电板、保护板、温度保护方案 | 验证充电和低电保护 | 样机阶段可用模块，量产再集成 |
| 电量计模块 | MAX17048 / BQ274xx 同级模块 | 电量百分比估算 | 低电提醒和续航测试需要可靠电量数据 |

### 2.2 P0：必须准备的调试与测试工具

| 类别 | 建议准备 | 用途 |
| --- | --- | --- |
| USB-TTL 串口模块 | 2-3 个 | 查看 GNSS NMEA、模组 AT 指令、固件日志 |
| ST-Link 或 J-Link | 按 MCU 选型准备 | 固件烧录和调试 |
| 万用表 | 1 个 | 基础电压、电流、连线排查 |
| 可调直流电源 | 1 台 | 模拟电池电压、稳定供电 |
| 功耗分析工具 | Nordic Power Profiler Kit II 或同级工具 | 测量待机、定位、上报、寻宠模式功耗 |
| 逻辑分析仪 | 1 个 | 调试 UART/I2C/SPI 时序 |
| Android 测试机 | 1-2 台 | App 调试、推送、地图、蓝牙可选能力 |
| iPhone 测试机 | 1 台 | iOS App 调试、定位权限、推送 |
| 移动电源 | 1-2 个 | 外场测试补电 |

### 2.3 P1：工程样机阶段准备

当 P0 开发板链路跑通后，再进入工程样机。

| 类别 | 建议准备 | 用途 |
| --- | --- | --- |
| 自研 PCB | 2-3 版小批量打样 | 集成 Cat 1 bis、GNSS、MCU、IMU、电源、充电 |
| 3D 打印外壳 | 多个尺寸版本 | 验证项圈安装、佩戴姿态、天线位置 |
| 项圈样品 | 不同宽度、材质、尺寸 | 验证外挂和内置结构 |
| 防水材料 | 密封圈、点胶、超声焊接样件或临时密封材料 | 做初步防水验证 |
| 磁吸充电结构 | 磁吸触点、充电底座样件 | 验证防水和充电便利性 |
| 蜂鸣器/LED | 小型蜂鸣器、单色/双色 LED | 寻宠提示和设备状态提示 |

### 2.4 暂时不建议先准备的硬件

| 硬件 | 暂缓原因 |
| --- | --- |
| RTK 厘米级定位模块 | 成本、功耗、天线和服务依赖都不适合 MVP |
| 极致小型化猫用 Mini 模具 | MVP 应先验证定位和续航，再做极限结构 |
| 大规模量产工装 | P0/P1 阶段硬件方案还会变化 |
| 完整 IP67 量产结构 | 可以先做简化防水验证，等 PCB 和天线稳定后再投入 |
| eSIM 批量烧录工具 | 小样机阶段可先用 nano SIM 或少量测试 eSIM |

## 3. 除硬件之外，可以写代码的部分

### 3.1 可以立即开始的代码

这些代码不需要等真实硬件全部到位。

| 代码部分 | 建议目录 | 说明 | 是否依赖真实硬件 |
| --- | --- | --- | --- |
| 设备模拟器 | `simulator/` | 模拟定位器按协议上报位置、电量、在线状态、告警事件 | 不依赖 |
| 云端 API | `server/` | 用户、宠物、设备、定位点、轨迹、围栏、告警接口 | 不依赖 |
| 数据库迁移 | `server/migrations/` | 建 users、pets、devices、location_points、geofences、alerts 等表 | 不依赖 |
| 轨迹处理算法 | `server/location/` | 定位点校验、去重、抽稀、平滑、轨迹查询 | 不依赖 |
| 电子围栏算法 | `server/geofence/` | 圆形围栏判断、离区防误报、回家提醒 | 不依赖 |
| App 原型 | `app/` | 地图首页、设备绑定、轨迹回放、安全区域、提醒页面 | 可先接模拟数据 |
| API 契约 | `docs/api/` 或 `server/openapi.yaml` | 定义设备上报协议和 App API | 不依赖 |
| 推送通知封装 | `server/notifications/` | 离区、低电、离线通知 | 可先用本地或测试推送 |
| 基础运维脚本 | `scripts/` | 创建设备、绑定设备、导入模拟轨迹、清理测试数据 | 不依赖 |
| 自动化测试 | `tests/` | API、围栏、轨迹算法、设备模拟器测试 | 不依赖 |

### 3.2 固件代码

固件代码可以先写框架，再逐步接入真实模组。

| 固件模块 | 内容 | 阶段 |
| --- | --- | --- |
| 设备状态机 | boot、idle、tracking、guard、lost、low_battery、ota | 立即开始 |
| 模组驱动适配层 | GNSS、Cat 1 bis、IMU、电量计、LED、蜂鸣器 | 拿到开发板后完善 |
| 数据上报协议 | 位置点、电量、在线状态、设备事件 | 立即开始 |
| 低功耗策略 | 静止待机、运动唤醒、低电降频、寻宠高频 | 开发板功耗测试后调参 |
| 离线缓存 | 网络失败时本地队列，恢复后补传 | 立即开始 |
| OTA 框架 | 固件版本、下载、校验、回滚 | MVP 可先做基础版 |
| 日志系统 | 串口日志、关键错误码、远程日志采样 | 立即开始 |

建议固件先抽象出统一接口：

```text
gnss_read()
modem_connect()
modem_publish()
imu_read_motion()
battery_read()
storage_enqueue()
storage_flush()
```

这样即使后续更换具体模组，也不会大面积重写业务状态机。

### 3.3 云端代码

云端是 MVP 中最适合先行开发的部分。

必须先写：

- 设备接入鉴权。
- 定位点上报接口。
- 最新位置查询接口。
- 历史轨迹查询接口。
- 宠物与设备绑定接口。
- 圆形电子围栏配置接口。
- 离区、低电、离线告警逻辑。
- WebSocket 或 SSE 实时位置推送。

建议技术拆分：

```text
server/
  api/              App API
  ingest/           设备上报接入
  location/         定位点处理、轨迹处理
  geofence/         围栏判断
  alerts/           告警生成和状态管理
  devices/          设备注册、绑定、在线状态
  pets/             宠物资料
  notifications/    推送通知
  migrations/       数据库迁移
```

### 3.4 用户 App 代码

App 可以先用模拟器和云端假数据开发。

必须先写：

- 登录/注册。
- 添加宠物。
- 绑定设备。
- 地图首页。
- 宠物当前位置标记。
- 设备电量、在线状态、定位精度展示。
- 历史轨迹回放。
- 安全区域设置。
- 消息提醒列表。
- 寻宠模式入口。

建议 App 的早期原则：

- 先打通真实使用流程，不急着做复杂视觉。
- 地图首页必须优先。
- 定位精度和定位来源必须展示，避免用户误解。
- 所有页面都要支持“设备暂无数据”的空状态。

### 3.5 设备模拟器

设备模拟器建议尽早写，因为它能让云端和 App 在真实硬件到位前开发。

模拟器需要支持：

- 按固定路线生成定位点。
- 模拟电量下降。
- 模拟在线、离线、重连。
- 模拟离开电子围栏。
- 模拟寻宠模式高频上报。
- 模拟 GNSS 漂移点和低精度点。
- 模拟离线缓存后批量补传。

模拟数据示例：

```json
{
  "device_id": "dev_mvp_001",
  "ts": 1782432000,
  "lat": 31.2304,
  "lng": 121.4737,
  "accuracy_m": 12,
  "source": "gnss",
  "battery_pct": 78,
  "motion_state": "moving",
  "event": "location"
}
```

## 4. MVP 推荐并行开发方式

### 第 1 阶段：没有真实硬件时

可以并行做：

1. 云端基础 API。
2. 数据库模型。
3. 设备模拟器。
4. App 地图首页和轨迹页。
5. API 契约。
6. 固件状态机框架。

目标：用模拟器跑通“位置上报 -> 云端存储 -> App 展示”。

### 第 2 阶段：开发板到位后

可以并行做：

1. 固件接入 GNSS。
2. 固件接入 Cat 1 bis 网络。
3. 固件接入 IMU。
4. 固件真实上报云端。
5. 云端处理真实设备数据。
6. App 显示真实设备位置。

目标：用开发板跑通真实定位和真实上报。

### 第 3 阶段：工程样机到位后

可以并行做：

1. 外场轨迹测试。
2. 功耗测试。
3. 围栏误报优化。
4. 佩戴结构和天线位置优化。
5. App 寻宠模式优化。
6. 设备离线补传验证。

目标：验证真实宠物佩戴场景下的稳定性。

## 5. 首轮开发任务清单

### 5.1 硬件采购任务

1. 采购 Cat 1 bis 开发板。
2. 采购双频 GNSS 评估板。
3. 采购 MCU 开发板。
4. 采购 LIS2DW12 或同级 IMU 模块。
5. 采购 LTE/GNSS 天线和转接线。
6. 准备至少两个运营商的测试 SIM。
7. 准备 500-700mAh 锂电池、充电模块、电量计模块。
8. 准备串口、调试器、功耗分析工具、万用表、可调电源。
9. 准备项圈和简易外壳材料，用于外场佩戴测试。

### 5.2 代码开发任务

1. 定义设备上报协议。
2. 定义 App API。
3. 建立数据库模型。
4. 实现设备模拟器。
5. 实现定位点上报接口。
6. 实现最新位置查询。
7. 实现历史轨迹查询。
8. 实现圆形电子围栏。
9. 实现离区、低电、离线告警。
10. 实现 App 地图首页。
11. 实现 App 轨迹回放。
12. 实现 App 安全区域设置。
13. 实现固件状态机框架。
14. 实现固件数据上报协议封装。

## 6. 建议先不做的代码

| 代码 | 暂缓原因 |
| --- | --- |
| 管理端 Vision 完整后台 | 当前 MVP 先验证 C 端闭环 |
| B 端多组织权限 | 会显著增加复杂度 |
| 商城/会员/订阅支付 | 需要商业模式进一步确认 |
| 宠物健康分析 | 数据不足，容易偏离定位 MVP |
| 社区内容功能 | 与核心定位闭环无关 |
| 多边形复杂围栏 | MVP 先用圆形围栏验证告警能力 |
| 复杂热力图 | 轨迹数据量积累后再做 |

## 7. 最小可运行演示目标

第一版可运行演示不需要真实量产硬件，只要达到：

1. 设备模拟器持续上报一条宠物移动路线。
2. 云端保存定位点。
3. App 地图实时显示宠物位置。
4. App 可以回放历史轨迹。
5. 用户设置圆形安全区域。
6. 模拟器走出安全区域后，App 收到离区提醒。

这版演示完成后，再接真实开发板。这样可以把软件链路风险提前解决，硬件到位后只需要替换数据来源。
